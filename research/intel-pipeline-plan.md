# AI 行业情报采集流水线方案

日期：2026-07-20  
范围：调研与方案设计，不写实现代码。

## 目标

把“手动刷 X / 小红书收集 AI 行业情报”升级为一条可控流水线：

```text
用户意图
  “帮我盯着 AI Agent 行业最新动向”
        |
        v
主题配置 / 来源配置 / 风险边界
        |
        v
合规采集 / 辅助浏览 / 平台 AI 总结
        |
        v
去重 / 可信度评分 / 价值评分
        |
        v
技术 / 市场 / 竞品 / 融资 / 产品 / 风险 分层
        |
        v
决策简报 / 原始链接 / 待跟进问题 / 触发提醒
```

核心判断：这条能力不是“爬虫产品”，而是“情报助理”。底线是小规模、可解释、可追溯、尊重平台规则，不做规模化抓取和数据倒卖。

## 1. 采集技术路线

### 1.1 方案 A：调用平台自带 AI，让平台先总结

例子：

- X / Grok：让 Grok 对某个账号、主题、列表或时间窗口做总结，我们只采集摘要、链接和人工确认后的重点。
- 小红书“问一问”：利用平台自己的问答/总结入口，把站内笔记经验整理成答案。

优点：

- 成本最低：不用自己处理大量原始内容。
- 反爬压力最低：主要通过平台提供的交互入口获取结果。
- 信息密度高：平台 AI 已经做过一次粗筛和压缩。
- 合规相对更稳：比直接批量抓页面更接近正常用户使用。

缺点：

- 受平台限制：平台 AI 覆盖范围、排序逻辑、上下文窗口和回答质量不可控。
- 不透明：无法知道它漏掉了哪些内容，也难以复现。
- 不适合严肃留档：平台 AI 可能不给完整来源，或者来源链接不稳定。
- X / Grok 与小红书“问一问”都可能有账号、地区、订阅、灰度限制。

适用场景：

- 每日早报、粗粒度趋势判断、低成本探索。
- 作为第一层“情报雷达”，不是唯一证据源。

结论：

方案 A 应作为第一优先入口，尤其适合“普通人零门槛”。但它不能单独支撑决策闭环，必须保留原始链接、截图或人工可复核证据。

### 1.2 方案 B：OpenClaw 浏览器自动化辅助浏览

OpenClaw 官方文档显示，它可以运行隔离的 Chrome/Brave/Edge/Chromium profile，执行打开标签、读取页面、点击、输入、拖拽、截图、PDF 等浏览器自动化动作；也可以通过用户 profile 接入真实登录会话。这说明它适合做“浏览器里的助理”，而不是只做后台 API。

可行能力：

- 打开 X 搜索、列表、关注页、Grok、小红书搜索页。
- 按人类确认的主题词浏览结果。
- 读取可见页面文本、截图留证、保存链接。
- 对收藏、书签、已打开页面、用户手动选中的内容做整理。
- 在需要登录、验证码、风控确认时停下来，让用户接管。

关键限制：

- 不应把 OpenClaw 用作“绕过反爬”的工具。
- X 服务条款明确禁止未经许可的 crawling/scraping，X 自动化规则也提示不要使用非 API 的网站脚本化自动化。
- 小红书协议同样禁止非法抓取、模拟下载、通过未授权第三方软件使用平台或影响平台运行。
- 因此，OpenClaw 路线只能定位为“个人研究辅助浏览 / 人机协同整理”，不能定位为“无人值守规模化抓取”。

适用场景：

- 用户已登录、自己本来就会浏览的页面，让工作台帮忙做低频、慢速、可中断的阅读整理。
- 抓取对象是用户当前可见内容、手动打开内容、收藏/书签内容。
- 需要截图、人工确认、页面交互，而 API 拿不到。

结论：

方案 B 技术上最贴近“像真人浏览”，但合规风险最高。可作为个人版实验能力和人工辅助能力，不应作为默认采集主路径。

### 1.3 方案 C：第三方采集 API / 数据服务

可选方向：

- 官方 API：X API / xAI API / 平台开放接口。
- 商业数据服务：Bright Data、Apify 等提供 X/Twitter 数据采集或数据集 API。
- 行业聚合源：RSS、新闻 API、公开博客、GitHub Trending、Hacker News、Product Hunt、arXiv、论文/项目订阅。

优点：

- 工程最稳定：结构化 JSON、调度简单、失败可重试。
- 适合做 MVP：先验证“分层 + 简报”的产品价值。
- 可以控制频率、成本、去重和留档。

缺点：

- 成本可能高：X Enterprise 是定制价；第三方 scraper 按记录数、并发或计算量收费。
- 合规责任不能外包：第三方声称合规不等于我们一定合规，仍要审平台条款和使用场景。
- 数据范围有限：官方 API 有 tier、rate limit、读量和再分发限制；第三方服务可能失效或被平台封堵。
- 与“去第三方依赖”战略冲突：第三方服务适合早期验证，不适合作为长期核心依赖。

适用场景：

- MVP 验证、低风险公开数据、预算可控的定时采集。
- 需要可重复、可追溯、可结构化的来源。

结论：

短期推荐 C 中的“官方 API / 合规数据服务 + 自主本机代理封装”。长期目标是把供应商封装为可替换 adapter，不把任何第三方 scraper 变成核心依赖。

### 1.4 推荐路线

推荐采用“三层采集”：

1. 默认层：合规公开源和官方 API。
   - X API、xAI/Grok API、RSS、GitHub、论文、新闻、官方博客。
   - 负责稳定、可复现、可留档。

2. 辅助层：平台 AI 总结。
   - Grok、小红书“问一问”等。
   - 负责低成本发现主题和生成候选线索。

3. 人机协同层：OpenClaw 浏览器辅助。
   - 只做小规模、低频、用户账号下的可见内容整理。
   - 遇到登录、验证码、风控、付费墙、异常提示，一律停下让用户处理。
   - 不做规避检测、不做批量数据搬运。

不推荐第一版直接做“小红书/X 无人值守拟人化爬虫”。技术上可能能做一点，但产品上会把工作台带向高风险、低确定性的方向。

## 2. 反爬 / 拟人化：核心难点

### 2.1 平台如何识别机器

平台和反自动化系统通常不是靠单一信号判断，而是多层组合：

1. 网络层
   - IP 信誉、ASN、数据中心 IP、代理 IP、异常地理位置。
   - TLS / HTTP 指纹、请求头组合、协议栈特征。
   - 短时间内请求量、请求间隔、失败率、重试模式。

2. 浏览器与设备层
   - User-Agent、语言、时区、屏幕尺寸、字体、GPU、CPU 核心数、device memory。
   - WebGL / Canvas / Audio 等浏览器指纹。
   - Headless Chrome、Playwright、Selenium、Puppeteer stealth 等自动化痕迹。
   - Cookie、localStorage、登录状态、设备历史是否稳定。

3. 行为层
   - 页面跳转路径是否自然。
   - 停留时长、滚动节奏、点击分布、鼠标移动、键盘输入速度。
   - 是否只访问搜索结果和详情页、不看评论、不停留、不回退。
   - 是否固定时间、固定间隔、固定关键词、固定页面深度。

4. 账号层
   - 新号高频浏览、频繁搜索、跨地域登录。
   - 账号没有正常关注、收藏、点赞、停留历史。
   - 行为和账号画像不一致。
   - 多账号同质行为。

5. 内容与业务层
   - 只采集不消费。
   - 大量访问相似页面。
   - 命中特定敏感接口或异常参数。
   - 触发平台的内容安全、反垃圾、风控策略。

### 2.2 “像真人”不等于“规避风控”

这里必须划清边界：

- 可以做：慢速、低频、可中断、用户可见、尊重平台提示的辅助浏览。
- 不做：指纹伪造、验证码绕过、代理池轮换、批量账号、绕过登录/付费/权限、规避平台限制。

安全做法：

- 使用用户明确授权的专用账号，不使用主账号。
- 每次采集规模小：例如 20-50 条候选内容，而不是几千条。
- 时间窗口低频：每日 1-3 次，不做分钟级轮询。
- 采集只保存必要信息：链接、标题/正文摘要、作者公开名、时间、截图哈希、价值标签。
- 保留用户手动暂停、接管、删除历史的能力。
- 遇到验证码、异常登录、平台风控提示，立即停止，不自动尝试解决。
- 对平台 AI 或浏览器结果做“引用式摘要”，不复制大段原文，不重发布内容。

不建议写入产品的能力：

- “反检测模式”
- “绕过验证码”
- “自动养号矩阵”
- “代理池”
- “批量采集小红书/推文原文”
- “商业监控数据售卖”

这些会把产品从“个人情报助理”推向“规避平台规则的采集工具”，与工作台的长期信任不一致。

### 2.3 OpenClaw 的 Computer Use 能否胜任

能胜任的部分：

- 打开页面、滚动、点击、截图、抽取可见文本。
- 在隔离浏览器 profile 中执行，不污染用户日常浏览器。
- 对页面状态做观察，遇到 blocker 停下。
- 作为“慢速人机协同阅读器”，把用户原本要手动刷的动作半自动化。

不应承诺的部分：

- 稳定绕过 X / 小红书反爬。
- 大规模无人值守采集。
- 长期保持账号不被风控。
- 用鼠标轨迹模拟、指纹伪造等方式伪装真人。

结论：

OpenClaw 可以承担“人类浏览助理”，不能承担“反爬突破器”。第一版如果用 OpenClaw，必须限制在：小规模、低频、可见页面、用户授权、可暂停。

### 2.4 风险控制

账号策略：

- 专用账号，不使用核心生活/业务账号。
- 不批量注册，不买号，不共享账号池。
- 账号行为保持一致：关注 AI Agent、模型、自动化、创业、AI 产品相关账号和列表。
- 不自动发帖、私信、评论、关注、点赞，第一版只读。

频率策略：

- 每日固定 1-3 个采集窗口。
- 每个平台每次只采集几十条候选。
- 失败后退避，不连续重试。
- 周末/夜间可以少量运行，但不要做 24 小时持续轮询。

数据策略：

- 只保存摘要和链接，必要时保存截图作为个人备忘证据。
- 不保存非公开内容。
- 不保存敏感个人信息。
- 不对外提供原始数据下载。

产品策略：

- UI 上明确标注“个人研究辅助，不用于商业批量采集”。
- 对每个平台显示风险等级。
- 默认关闭高风险浏览器采集，需要用户手动启用。
- 把官方 API / RSS / 公开博客作为默认来源。

## 3. 算法杠杆利用：把推荐算法当免费情报员

核心思路：不是对抗推荐算法，而是训练它服务我们。

### 3.1 如何驯化推荐流

1. 建立专用账号画像
   - 只关注 AI Agent、AI 自动化、模型、开发者工具、企业软件、投资人、研究员、竞品官方账号。
   - 避免娱乐、无关热点、情绪化内容污染画像。

2. 持续正反馈高质量内容
   - 对高质量内容做人工确认：收藏、加入列表、停留阅读、打开原文链接。
   - 对低质量内容减少停留，不互动。
   - 不建议第一版自动点赞/关注，因为这属于平台写操作，风险更高。

3. 使用列表和搜索词约束推荐
   - X：维护 AI Agent 列表、创业者列表、研究员列表、竞品列表。
   - 小红书：维护关键词集合和收藏夹，尽量让平台搜索/推荐集中到 AI 工具、Agent、效率工具、个人自动化。

4. 反馈闭环
   - 简报里让用户标记“有价值 / 无价值 / 重复 / 噪音 / 需要跟进”。
   - 工作台把这些反馈转成下一轮关键词、账号权重、来源权重。

### 3.2 价值

- 平台推荐算法本身就是免费召回系统。
- 我们不需要一开始覆盖全网，只要把账号流训练成“AI Agent 情报流”。
- 长期看，用户越用，情报源越准，简报质量越高。

### 3.3 风险

- 过度训练会形成信息茧房。
- 推荐流会被热点、营销号、标题党污染。
- 平台算法变化会导致质量波动。

缓解：

- 保留固定可信源列表。
- 每周加入“外部冷启动源”：GitHub、论文、HN、Product Hunt、官方博客。
- 简报中明确区分“平台推荐发现”和“可信源确认”。

## 4. 分层与整合

### 4.1 原始信息对象

每条候选内容进入工作台后，先统一成结构：

```json
{
  "source": "x",
  "url": "https://x.com/...",
  "author": "公开作者名",
  "publishedAt": "2026-07-20T00:00:00Z",
  "collectedAt": "2026-07-20T00:00:00Z",
  "rawTextExcerpt": "短摘录，不保存大段全文",
  "summary": "一句话摘要",
  "topic": ["AI Agent", "browser automation"],
  "category": "technical",
  "valueScore": 0.82,
  "noiseScore": 0.12,
  "evidence": {
    "link": true,
    "screenshot": "optional-local-path",
    "sourceType": "official_api / platform_ai / assisted_browsing / rss"
  }
}
```

原则：

- 保留链接和证据，不把平台内容搬成本地数据库。
- 摘录只用于识别和复核，简报以转述和分析为主。
- 所有自动判断都要能追溯到来源。

### 4.2 分层过滤

第一层：硬过滤

- 语言：中文 / 英文。
- 时间：最近 24 小时 / 7 天。
- 主题词：AI Agent、browser automation、computer use、OpenClaw、Hermes、Claude Code、Codex、Coze、Grok、MCP、workflow automation。
- 排除词：抽奖、无关营销、纯情绪、无来源搬运、币圈广告。

第二层：去重

- URL 去重。
- 文本相似度去重。
- 同一新闻多账号转发聚合为一个事件。
- 原始发布优先，二手转述降权。

第三层：价值评分

高价值信号：

- 官方发布、产品更新、论文、开源仓库、融资/收购、价格/API/政策变化。
- 有截图、链接、数据、代码、demo、benchmark。
- 被多个可信账号独立提到。
- 与工作台战略相关：Agent runtime、Computer Use、本机代理、平台品控、零门槛、模型分层调用。

低价值信号：

- 只有观点没有证据。
- 标题党、重复搬运、AI 生成味重。
- 纯转发无新增信息。
- 与产品路线无关。

第四层：分类

- 技术：模型、Agent runtime、browser automation、Computer Use、MCP、工具链。
- 市场：用户需求、价格变化、商业模式、行业采用。
- 竞品：豆包、扣子 Coze、飞书、Hermes、OpenClaw、Grok、Claude、Codex、Perplexity 等。
- 合规/平台：API 政策、反爬、账号治理、数据授权、模型下线。
- 机会：可做功能、可借鉴交互、可切入垂直场景。
- 风险：被大厂覆盖、依赖第三方、成本不可控、平台封禁。

### 4.3 简报结构

每日简报建议结构：

```text
# AI Agent 行业情报简报

日期：
覆盖来源：
总体判断：

## 1. 今天最重要的 3 件事
1. 事件 / 来源 / 为什么重要 / 对工作台影响
2. ...

## 2. 技术动向
- 新模型 / 新 Agent 能力 / 新开源项目 / benchmark

## 3. 竞品动向
- 大厂 / 独立产品 / 桌面 Agent / 浏览器 Agent

## 4. 市场与用户信号
- 用户痛点 / 付费信号 / 讨论热度 / 典型观点

## 5. 风险与合规
- API 政策变化 / 平台限制 / 成本变化 / 下线风险

## 6. 对 AI Workbench 的建议
- 立即做
- 观察
- 不做

## 7. 原始来源
- 链接列表，按价值排序
```

决策价值来自最后一节：“对 AI Workbench 的建议”。情报不是为了看热闹，而是为了指导产品取舍。

## 5. 合规边界

### 5.1 风险说明

X：

- X 服务条款明确禁止未经书面许可的 crawling / scraping。
- X 自动化规则提示不要使用非 API 的网站脚本化自动化，否则可能导致账号永久封禁。
- X API 有开发者协议、用途披露、rate limit、再分发限制和商业使用限制。

小红书：

- 小红书用户协议禁止非法抓取、模拟下载、盗取图文/视频/用户信息。
- 小红书开放平台开发者协议要求仅获取功能必要的数据，不得转移、出售或用于特定应用之外。
- 小红书对 AI 托管类账号和模拟真人互动有治理倾向，自动发布、自动互动尤其高风险。

第三方采集服务：

- Bright Data、Apify 等能提供结构化采集能力，但使用者仍需承担合法、合规、用途适配责任。
- 第三方服务适合早期验证，不适合成为核心不可替代依赖。

### 5.2 个人研究安全做法

可以做：

- 使用官方 API、RSS、公开新闻、官方博客、GitHub 等合规来源。
- 对自己账号正常可见的内容做小规模阅读辅助。
- 保存链接、摘要、标签和少量截图作为个人备忘。
- 不发布、不私信、不评论、不影响平台秩序。
- 尊重 robots、API rate limit、平台提示和账号风控。
- 给用户明确删除数据和关闭采集的入口。

谨慎做：

- 使用 OpenClaw 浏览器辅助浏览 X / 小红书。
- 使用第三方 scraper 服务抓公开内容。
- 使用平台 AI 总结结果作为决策输入。

不能做：

- 规模化商用采集。
- 倒卖数据、输出原始数据集、提供数据下载 API。
- 抓取非公开、登录后敏感、付费墙、私信、评论区个人信息。
- 绕过验证码、登录限制、风控、访问控制。
- 批量注册账号、使用账号池、代理池。
- 自动发帖、评论、私信、关注、点赞来操纵推荐或互动。

## 6. 最小可用版：X + AI Agent 单主题

### 6.1 MVP 目标

用户说：

> 帮我盯着 AI Agent 行业最新动向。

工作台每天产出一份简报：

- 20-50 条候选内容。
- 去重后 5-10 条高价值线索。
- 按技术 / 竞品 / 市场 / 风险分类。
- 输出 3 条最重要判断和对 AI Workbench 的建议。

### 6.2 推荐 MVP 路线

第一版推荐不从小红书开始，而从 X 开始。

原因：

- AI Agent 行业的一手信息更集中在 X、GitHub、官方博客、论文和英文社区。
- X 有官方 API 和大量第三方结构化服务可选，虽然成本和协议约束较强，但工程验证更直接。
- 小红书更适合验证中文用户需求、工具体验、消费侧反馈，适合作为第二平台。

MVP 采集组合：

1. X 官方 API / 合规数据服务
   - 搜索关键词：`AI Agent`、`browser agent`、`computer use`、`OpenClaw`、`Hermes`、`Claude Code`、`Codex`、`Coze`、`Grok`、`MCP`。
   - 账号列表：竞品官方、核心开发者、投资人、研究员。
   - 保存链接、作者、时间、短摘录、互动指标。

2. Grok / xAI 辅助总结
   - 对同一主题做平台内总结或通过 xAI API 做二次压缩。
   - 只把它当召回和压缩层，不当事实来源。

3. OpenClaw 人机协同
   - 打开 X 列表和搜索页。
   - 用户登录和风控处理由用户完成。
   - OpenClaw 只读取可见内容、截图、保存链接。
   - 默认慢速、小规模、只读。

### 6.3 工作量预估

只做方案验证，不接 UI：

- 来源配置：0.5 天。
- X API / 数据服务 adapter 评估与手工样例：0.5-1 天。
- 原始信息 schema、去重、评分 prompt：1 天。
- 简报模板和生成链路：0.5 天。
- 本地文件留档和人工验收：0.5 天。

合计：2.5-3.5 天。

接入工作台聊天入口：

- “帮我盯着 AI Agent 行业最新动向”意图识别：0.5 天。
- 后台任务状态、简报展示、来源链接：1 天。
- 调度、失败重试、限速、配置页：1-2 天。

合计：4.5-6.5 天。

如果第一版用 OpenClaw 浏览器辅助替代 API：

- 额外增加 2-4 天调试页面状态、登录态、截图、滚动、失败恢复。
- 合规和封号风险更高。
- 不推荐作为第一条主路径。

### 6.4 MVP 验收标准

输入：

- 主题：AI Agent。
- 平台：X。
- 时间窗口：最近 24 小时。
- 候选上限：50 条。

输出：

- `research/intel-digests/YYYY-MM-DD-ai-agent.md`
- 至少 10 条来源链接。
- 至少 5 条去重后的高价值线索。
- 每条线索有分类、价值理由、对工作台影响。
- 简报最后必须给出“做 / 观察 / 不做”建议。

质量门槛：

- 重复内容少于 20%。
- 明显广告/噪音少于 20%。
- 每个高价值判断至少有一个来源链接。
- 不保存大段原文，不导出数据集。

## 7. 推荐落地顺序

1. 先做“情报简报格式”和人工导入样例。
   - 手动给 20 条 X 链接，验证分层、评分、简报是否有决策价值。

2. 接官方 API / 合规数据源。
   - 先跑 X 单主题。
   - 不急着接小红书。

3. 加 OpenClaw 辅助浏览。
   - 只读、小规模、人工可接管。
   - 作为补充来源，不做默认无人值守采集。

4. 加推荐算法训练机制。
   - 专用账号、专用列表、人工反馈。
   - 不做自动互动写操作。

5. 扩展到小红书。
   - 重点看中文用户需求、AI 工具真实使用反馈、非技术用户门槛。
   - 第一版优先利用“问一问”或人工选中内容整理，不做批量抓取。

## 8. 最终建议

这条能力值得做，但不要把它做成“爬虫系统”。正确定位是：

> AI Workbench 的行业情报员工：合规采集公开线索，辅助用户阅读平台内容，把噪音压缩成决策简报。

第一版建议：

- 主路径：X 官方 API / 合规数据服务 + 本机模型代理总结。
- 辅助路径：Grok / 小红书问一问等平台 AI。
- 实验路径：OpenClaw 浏览器辅助，只做个人研究、小规模、只读、人工可接管。

这样既能解决“手动刷信息效率低”的问题，又不把产品带进高风险反爬对抗。
