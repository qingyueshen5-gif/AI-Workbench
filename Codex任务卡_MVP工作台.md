# 任务卡：AI Workbench MVP 开发

> 交付方式：直接在现有 GitHub 仓库里开发，不需要额外确认设计思路，按此文档执行即可。
> 背景文档：README.md / PRODUCT.md / PRINCIPLES.md / CURRENT_TASK.md（仓库中已有，若与本文档冲突，以本文档为准，因为这是最新对齐结果）

---

## 一、目标（本次任务范围）

跑通一个最小闭环：**发指令 → 系统留痕 → 展示进度**。

**不做**：自动执行任务、多 Agent 调度、用户登录系统、真实数据库、AI 自动调用。
**只做**：一个能本地跑起来的网页，四个模块，手动操作即可，先让人（也就是李）自己每天用起来。

---

## 二、技术选型（不用另外讨论，直接按这个来）

- 前端：React + Tailwind
- 存储：本地 JSON 文件 或 SQLite，不需要远程数据库，不需要用户账号系统
- 单机跑通即可，不考虑多设备同步

---

## 三、四个模块（具体功能）

### 1. 首页
- 显示"今天目标"（一行文字，手动输入编辑）
- 显示"今日完成"清单（可勾选的任务项）
- 显示完成进度百分比（已完成数 / 总数）

### 2. 聊天页
- 一个输入框，可以发消息
- 消息记录列表（按时间顺序，显示时间戳+内容）
- 每条消息可以手动标记为"这是一个任务"，标记后自动同步进任务列表

### 3. 任务状态页
- 任务列表，每项显示状态：待开始 / 进行中 / 已完成 / 失败
- 点开任务能看到：负责人（GPT / Codex / Claude / 人工，手动选择）、创建时间、备注
- 状态可以手动切换

### 4. 历史记录页
- 按天归档，展示过去每天的"目标 + 完成情况"
- **关键规则**：任何任务被标记为"失败"时，必须强制填写一句"失败原因"才能保存
  - 这是"错误记忆库"的雏形：以后要能按关键词搜索历史失败原因，避免重复踩坑
  - 现阶段只需要能存、能搜索（哪怕是最简单的文本匹配）即可，不需要 AI 自动分析

---

## 四、关于"安全确认"（这次先用最简形式，不用做复杂系统）

- 目前不接入任何自动执行能力，所以暂时不需要真正的风险分级逻辑
- 唯一需要做的：**"失败原因必填"**这一条规则本身，就是本次唯一的"强制确认"机制
- 更复杂的授权中心（哪些操作自动/哪些需确认/付款限额等）留到后续版本，本次不做，避免范围蔓延

---

## 五、验收标准

- 李本人能连续使用 3-7 天：每天打开网页写目标、勾任务、必要时标记失败原因
- 到期后回答一句：好不好用？卡在哪？——这个反馈本身，就是下一轮迭代的输入

---

## 六、给 Codex 的执行提示

- 优先跑通闭环，不要在样式、动画、扩展功能上花时间
- 遇到设计上的模糊点，按"最简单能用"的方向做决定，不需要反复确认
- 完成后在 CURRENT_TASK.md 里更新进度，并说明改了哪些文件、怎么验证功能正常
