# 任务：产品定位修正与判断依据文档

日期：2026-07-24

状态：completed

## 目标

修正当前状态文档中过度绑定 DeepSeek 的表述，明确 AI Workbench 的产品定位是模型与 Agent 无关的调度框架。同时新增判断依据文档，让新接手的 AI 理解决策背后的原因。

## 发现的表述偏差

- `PRODUCT.md` 缺少最前置的模型无关定位声明。
- `CONTEXT.md` 的当前架构直接写成通向 DeepSeek 官方 API，容易被误读为 DeepSeek 客户端。
- `CONTEXT.md` 的已完成能力把“无需用户填写 DeepSeek API Key”作为产品口径，没有说明 DeepSeek 只是当前生产 provider。
- `CURRENT_PROGRESS_AUDIT.md` 的已完成能力和风险描述把 DeepSeek 写得过于中心化，缺少“当前实现而非定位”的边界。
- `AI-Workbench-Handoff.md` 的自动快照和交接规则没有提示新对话阅读判断依据文档。
- `EXECUTION_PROTOCOL.md` 的新对话交接规则仍是旧的两份文件口径，没有把判断依据纳入默认上下文。

## 修正内容

- 在 `PRODUCT.md` 顶部加入模型与 Agent 无关的调度框架定位声明。
- 将当前状态文档统一为“当前生产实现为 DeepSeek，架构支持多 provider”的口径。
- 在 `EXECUTION_PROTOCOL.md` 和 `AI-Workbench-Handoff.md` 写明：新对话如需理解决策背景，应阅读 `THINKING.md`。
- 将新对话默认提供文件更新为 `AI-Workbench-Handoff.md`、`NEXT_STEP.md`、`THINKING.md` 三份；判断验收时再补对应 `verification/<task>/summary.json`。
- 更新 Handoff 自动生成模板，避免刷新后恢复旧口径。

## THINKING.md 用途和内容

`THINKING.md` 用于记录产品负责人判断依据，避免新对话只看到结论而无法延续判断方式。内容按“主张 + 为什么”记录：

- 为什么坚持一个输入框。
- 为什么零门槛不能因成本妥协。
- 为什么走胖东来路线。
- 为什么知名度优先于赚钱。
- 为什么野性用在产品定位而不是规则上。
- 为什么是框架而不是模型。
- 为什么产品效果与用户水平正相关。
- 为什么执行能力借用生态而不自建。
- 为什么决策人的状态是关键输入。
