# PRINCIPLES.md — 设计原则

## 1. 为自己而做

不是做一个"给别人看"的产品。每个功能都解决我自己的真实痛点。

## 2. 简单胜过完备

一个能做一件事的脚本，胜过十个"支持所有场景"的框架。

```
好: python summarize.py article.txt
坏: python main.py --mode=summarize --input=article.txt --output=result.json --format=markdown --model=gpt-4
```

后台可以复杂，用户面前必须简单。用户只表达目标，系统负责拆解、选择模型、调用工具、检查、修复和交付。

## 3. 文本即接口

输入是文本，输出是文本。不定义新格式，不造新协议。纯文本天然可以被任何工具消费。

## 4. 先跑起来再优化

不要"设计好了再动手"。用最糙的方式先跑通，用了几天觉得哪里不爽再改。

## 5. 本地优先，隐私第一

- 用户数据、任务状态和执行证据本地优先保存
- 真实模型 key 不进入安装包、前端、员工配置或用户电脑
- 模型请求可以通过 AI Workbench 自控 Managed Proxy 调用
- 不把用户数据上传到无关第三方服务

## 6. 可还原

- 每个功能有清晰的输入 → 输出映射
- 同样的输入，同样的输出
- 出错了能复现

## 7. 真实完成

- 不把“说了怎么做”当成“做完了”。
- 关键动作要有证据、状态和失败原因。
- 失败时给中文人话解释，能恢复就恢复，不能恢复就说清阻塞点。

## 8. 高质量、快速、低损耗

- 结果必须简单、清楚、可执行。
- 避免不必要的模型调用、Token 浪费和重复劳动。
- 能用便宜稳定路径解决的，不默认上复杂路径。
- 质量检查、成本控制和执行证据必须属于工作台控制层。

## 9. 透明可追溯

- README.md 让人知道这是什么
- 代码里的注释解释为什么这样做
- 不写"完整文档"——没人看
- 当前进度、历史决策、验收证据和下一步必须能被新对话读取并接手。
