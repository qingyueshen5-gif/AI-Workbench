# AI Workbench 底层架构设计

> 本文是工作台接入多模型、多 Agent、多工具前的底层框架约束。后续接 Hermes、OpenClaw、Codex、Claude Code、GPT、Gemini 等，都必须先符合这里的规则，不能临时缝线。

---

## 一、产品第一原则

用户没有技术背景，只要结果。

"人人都能用"决定产品上限，不是技术多牛决定产品上限。参照炒股：背后交易系统、行情源、风控、撮合再复杂，用户只面对"买/卖"两个动作，赚不赚看判断力。

工作台也一样：水下可以很复杂，但水面上必须极简单。用户只做一件事：对结果说"行"或"不行"。

所以工作台的职责不是把更多 AI、更多 Agent、更多配置暴露给用户，而是把它们全部收进水下，变成可调度的员工。用户只跟工作台聊天框说目的，工作台负责拆解、调度、执行、验证、兜底和汇报。

判断每个设计是否合格，只看一句话：

任何让用户去配环境、看报错、敲命令、按英文确认框，或者对技术细节做选择的设计，都是产品失败。

## 二、三条硬约束

### 1. 用户只判断结果，不判断过程

用户不能被要求判断技术过程。

禁止出现的问题：

- "Would you like to run Get-CimInstance..."
- "查这个数据有两个办法：快但可能不准 / 慢但准确，选哪个？"
- "要不要切换模型？"
- "要不要安装依赖？"
- "要不要重试 502？"

这些都不是用户该判断的事。工作台必须自己拿主意。

默认策略：

- 准确性优先，除非用户明确说赶时间
- 稳定性优先，除非用户明确说只要草稿
- 可恢复优先，除非用户明确允许冒险
- 中文大白话优先，不出现技术术语和英文确认框

只有这些情况才问用户：

- 花钱
- 删除数据
- 发消息给别人
- 修改账号、安全、权限设置
- 不可逆操作
- 法律、医疗、财务等高风险判断

即使必须问，也只能问中文大白话。例如：

- "这一步会删除原文件，删掉后不一定能恢复。要继续吗？"
- "这一步需要你登录账号。点这里打开登录页面。"

### 2. 成本与结果成正比，自动算账，不问用户

用户不需要知道这次用了哪个模型，也不需要被问"要用便宜的还是贵的"。

工作台自己算账：

- 量大、简单、重复的活，优先派便宜模型或本地工具
- 关键判断、质量要求高的活，自动派更强模型
- 需要浏览器、文件、系统操作的活，派对应 Agent
- 失败一次后，自动评估是重试、换模型、换 Agent，还是降级方案

用户永远不需要知道这次用了谁、花了多少，但系统必须留痕，用户想查时可以看到：

- 用了哪个模型或 Agent
- 做了哪些步骤
- 花了多少钱或 token
- 为什么选择它
- 失败和重试记录

### 3. 环境自愈层必须存在

不能假设用户会配置环境、看得懂报错、会去终端敲命令。

这些问题都属于工作台要消化的地基层问题：

- 代理断线
- 502
- API 临时失败
- 缺依赖
- 缺工具
- 登录过期
- 路径不对
- 权限错误
- Agent 掉线
- 配置迁移

正确做法：

- 能自动重连就自动重连
- 能换节点就换节点
- 能重试就重试
- 能自动安装依赖就安装
- 能自动修配置就修配置
- 登录过期时给用户一个"点这里登录"按钮
- 报错必须翻译成人话，并给一个可点击的解决动作

禁止把原始错误直接甩给用户。例如不能让用户看到一大段 traceback、502、permission denied、readonly database 后自己判断。

## 三、7 层架构

### 第 1 层：聊天入口层

这是用户唯一看到的主界面。

职责：

- 接收用户目的
- 展示结果
- 让用户判断"行"或"不行"
- 必要时用中文大白话请求确认

不做：

- 不展示模型选择器
- 不展示 Agent 选择器
- 不展示英文技术确认框
- 不要求用户填复杂表单

### 第 2 层：调度层

这是工作台的大脑。

职责：

- 判断用户意图
- 判断任务类型
- 判断风险等级
- 判断该派哪个员工
- 判断是否需要拆任务
- 判断失败后怎么兜底

调度逻辑必须放在工作台，不放在 Hermes、DeepSeek 或任何单个 Agent 里面。

调度顺序：

1. 先用明确规则判断明显任务
2. 不明确时，用模型输出结构化判断
3. 低置信度时，工作台自己选择更稳妥方案
4. 只有触发高风险边界，才问用户

### 第 3 层：员工接入层

所有模型和 Agent 都是员工。每个员工必须按同一套接口接入。

员工类型包括：

- 模型：DeepSeek、GPT、Claude、Gemini 等
- Agent：Hermes、OpenClaw、Codex、Claude Code 等
- 工具：搜索、文件、浏览器、终端、数据库等

统一接口：

- `healthCheck()`：员工现在能不能用
- `canHandle(task)`：是否适合这个任务
- `execute(task, context)`：执行任务
- `status(runId)`：查询进度
- `cancel(runId)`：取消任务
- `verify(result)`：验证结果
- `normalizeError(error)`：把原始错误翻译成工作台错误

员工档案示例：

```json
{
  "id": "hermes",
  "type": "agent",
  "name": "Hermes",
  "capabilities": ["browser", "terminal", "web_search", "memory"],
  "riskLevel": "medium",
  "costLevel": "medium",
  "healthCheck": "hermes --version",
  "adapter": "adapters/hermes"
}
```

核心代码只认识统一接口，不直接认识某个具体员工。以后接新员工，只新增 adapter 和员工档案，不改核心调度逻辑。

可借鉴成熟模式：

- MCP 的 tools/resources/prompts 暴露方式：适合做员工能力注册标准
- 插件系统：员工像插件一样注册能力，核心只调接口
- 适配器模式：每个员工内部怎么调用可以不同，对工作台暴露统一格式

### 第 4 层：任务引擎层

大任务必须被工作台拆成小任务，而不是直接扔给某个 Agent。

职责：

- 创建任务
- 拆解任务
- 管理依赖关系
- 分派员工
- 并行执行
- 收集结果
- 汇总结果
- 失败重试
- 失败换人
- 写入历史

任务流示例：

用户说："帮我把工作台接入 Hermes。"

工作台拆成：

1. 读取当前项目状态
2. 检查 Hermes 是否可用
3. 设计接入方案
4. 写代码
5. 验证 Hermes 调用
6. 记录结果
7. 汇报给用户判断

MVP 阶段先不做复杂工作流引擎，先用简单任务表和状态机。后续可以借鉴：

- LangGraph：适合多步骤、多 Agent、可恢复的 Agent 工作流
- Temporal：适合长期运行、定时任务、失败恢复，但 MVP 阶段偏重，后面再考虑

### 第 5 层：工作台记忆层

记忆必须归工作台所有，不能存在某个 Agent 肚子里。

记忆类型：

- 用户偏好
- 项目上下文
- 任务历史
- 错误经验
- 决策记录
- 环境状态

员工只能借阅记忆，不能各存各的。

执行流程：

1. 工作台从中央记忆库取出相关上下文
2. 工作台生成任务上下文包
3. 员工只拿到当前任务需要的记忆
4. 员工执行后可以提交记忆建议
5. 工作台审核后写入中央记忆库

高风险记忆必须用户确认后才能保存，比如账号、权限、长期偏好、自动执行习惯。

MVP 阶段用本地 JSON 或 SQLite。后续再加向量检索，用来按语义找历史经验。

### 第 6 层：验证与故障兜底层

员工不能自己宣布成功。必须由工作台验证。

不同任务要求不同证据：

- 代码任务：diff、测试结果、commit hash
- 文件任务：文件路径、内容摘要、读写结果
- 浏览器任务：截图、最终页面状态、关键字段
- 搜索任务：来源链接、日期、摘要
- 系统任务：命令结果、退出码、环境状态

故障处理：

- 报错：转成人话，内部保存原始日志
- 超时：自动停止，重试或换员工
- 掉线：自动重连
- 假完成：验证失败，不给用户报成功
- 能力缺失：换员工或降级处理
- 高风险操作：请求用户确认

用户看到的应该是：

"我尝试执行这一步时发现 Hermes 的日志权限有问题，但核心文件读取已经成功。这个权限问题已记录为后续环境任务。"

而不是：

`PermissionError: [Errno 13] Permission denied`

### 第 7 层：环境自愈层

这是和其他 6 层平级的基础能力，不是可有可无的辅助功能。

职责：

- 检查模型、Agent、工具是否可用
- 自动修复常见环境问题
- 自动重试临时故障
- 自动安装缺失依赖
- 自动迁移配置
- 自动续期或引导登录
- 把原始错误翻译成人话
- 给用户可点击解决按钮

典型处理方式：

| 问题 | 工作台动作 | 用户看到 |
|---|---|---|
| 代理断线 | 自动重连、换节点、重试 | "网络连接刚才不稳定，我已自动重试。" |
| 502 | 等待后重试，必要时换模型 | "服务临时不稳定，我换了备用方案继续处理。" |
| 缺依赖 | 自动安装并记录 | "缺少运行组件，我已安装并继续。" |
| 登录过期 | 打开登录按钮 | "需要重新登录，点这里登录后我继续。" |
| 权限错误 | 尝试修权限或换路径 | "本地权限异常，我正在尝试修复。" |
| 员工不可用 | 换候选员工 | "原执行工具不可用，我已换备用工具处理。" |

环境自愈层必须有自己的状态记录，不能只靠临时日志。这样同一个坑下次能自动识别。

## 四、四张核心表

MVP 阶段可以先存在 `data/workbench.json`。结构稳定后再迁移 SQLite。

### 1. `agents`：员工注册表

记录谁能干什么、怎么调用、是否健康。

关键字段：

- `id`：员工 ID，例如 `deepseek`、`hermes`
- `type`：`model` / `agent` / `tool`
- `name`：显示名称
- `capabilities`：能力列表
- `adapter`：调用适配器
- `riskLevel`：风险等级
- `costLevel`：成本等级
- `status`：`available` / `degraded` / `unavailable`
- `lastHealthCheckAt`：上次健康检查时间
- `failureCount`：近期失败次数
- `notes`：环境说明

### 2. `tasks`：任务表

记录用户目标和工作台拆出来的小任务。

关键字段：

- `id`
- `parentTaskId`
- `userGoal`
- `title`
- `status`：`pending` / `running` / `blocked` / `failed` / `done`
- `priority`
- `riskLevel`
- `assignedAgentId`
- `dependencies`
- `evidenceRequired`
- `createdAt`
- `updatedAt`
- `userVisibleSummary`

### 3. `runs`：执行记录表

记录每次员工执行的过程、结果、证据、失败和重试。

关键字段：

- `id`
- `taskId`
- `agentId`
- `status`
- `input`
- `output`
- `evidence`
- `errorRaw`
- `errorUserMessage`
- `retryCount`
- `costEstimate`
- `startedAt`
- `finishedAt`
- `verified`
- `verificationResult`

### 4. `memories`：工作台记忆表

记录工作台自己的长期记忆。

关键字段：

- `id`
- `type`：`preference` / `project_context` / `decision` / `error_lesson` / `environment`
- `content`
- `source`
- `confidence`
- `visibility`：是否给用户展示
- `requiresUserApproval`
- `approved`
- `createdAt`
- `updatedAt`

记忆写入规则：

- 项目事实可以自动写
- 错误经验可以自动写
- 临时任务上下文可以自动写
- 长期偏好要谨慎写
- 账号、权限、安全相关必须用户确认

## 五、分阶段实施顺序

### MVP 必须做

目标：先跑通"工作台调度一个员工，并验证结果"。

1. 新增 `ARCHITECTURE.md`，固定设计边界
2. 建立 `agents/tasks/runs/memories` 四类数据结构
3. 先注册两个员工：DeepSeek 和 Hermes
4. 做最小调度规则：普通回答给 DeepSeek，浏览器/文件/终端执行给 Hermes
5. 做 Hermes adapter：只支持最小命令调用和结果收集
6. 做中央记忆读取：把 `CONTEXT.md`、`VISION.md`、`CURRENT_TASK.md` 纳入工作台上下文
7. 做验证要求：执行任务必须返回证据
8. 做错误翻译：不把原始错误直接展示给用户
9. 做环境自愈最小版：检测 Hermes 是否可用，失败时给中文解释和内部记录

MVP 验收场景：

用户在工作台聊天框说："让 Hermes 读取 CURRENT_TASK.md 并总结今天待办。"

工作台应该：

1. 判断这是文件执行任务
2. 派 Hermes
3. 带上必要上下文
4. 让 Hermes 读取文件
5. 收到结果和证据
6. 写入任务历史
7. 用中文给用户展示结果
8. 不暴露命令、路径、报错细节，除非用户主动展开查看

### 第二阶段

目标：从单员工调度变成多员工协作。

1. 引入任务拆解
2. 支持多个候选员工排序
3. 支持失败后换员工
4. 接入 OpenClaw
5. 支持并行任务
6. 支持更完整的成本估算
7. 支持记忆建议和审核
8. 支持环境自愈按钮，例如"点这里登录"、"一键修复"

### 第三阶段

目标：长期运行和更强自动化。

1. 引入 LangGraph 或类似工作流编排
2. 评估 Temporal，用于长期任务、定时任务、失败恢复
3. 做云端部署
4. 做 7x24 在线 Agent
5. 做权限系统
6. 做安全员/监管 Agent
7. 做成本看板
8. 做环境问题自动归因和自动修复库

## 六、最终判断标准

架构是否成功，不看接了多少模型，也不看用了多少复杂框架。

只看用户体验：

- 用户是否只需要说目的
- 用户是否只需要判断结果
- 用户是否不用理解模型和 Agent
- 用户是否不用配环境
- 用户是否不用看英文报错
- 用户是否不用敲命令
- 工作台是否能自己选择、自己执行、自己验证、自己兜底

如果答案是是，这个架构才成立。
