# PRODUCT.md — 产品定义

## 愿景

做一个**自己每天都会用**、普通人也能直接使用的 AI 工作台。不做大而全，只做高频刚需；长期目标是让不同国家和地区的用户都能用一个输入框把事情办成。

## 目标用户定位

本产品只服务两种人，这是一切功能取舍的最高准绳：

- 普通人：不懂技术，只想要结果，一句话搞定。核心诉求是**零门槛**。
- 专业人：时间就是杠杆，用平台把效率拉满。核心诉求是**省时间**。

取舍铁律：任何新功能，先问“它是帮普通人拿结果，还是帮专业人省时间？”

两者都不是就不做。不追求吃下所有人，服务好这两类用户即可：普通人代表大多数，专业人代表高粘性和高回购。

长期用户范围是全球用户，不只服务中国或海外某一个市场。不同地区的模型、语言、支付、合规和平台规则可以分阶段适配，但产品入口必须保持一致：用户表达目标，工作台负责把目标变成结果。

## 核心体验：一个输入框

用户不需要理解模型、Agent、工具、脚本、上下文和错误恢复。用户只表达目标，后台承担：

- 读取上下文、长期目标、历史决策和当前进度；
- 拆解任务、选择模型、调用工具和分配执行者；
- 检查结果质量、发现失败、重试或解释原因；
- 留下证据，把最终结果交付给用户判断。

产品不是聊天壳，也不是工具列表。产品价值在于把复杂后台压成一个简单入口，把用户从配置、选择、重复解释和反复检查中解放出来。

## 产品边界

AI Workbench 可以借用 GPT、Claude、DeepSeek、Hermes、OpenClaw、浏览器自动化和其他成熟产品作为杠杆，但不能把核心控制权交出去。

必须由 AI Workbench 自己掌握：

- 任务状态；
- 长期记忆；
- 任务分配；
- 质量检查；
- 失败恢复；
- 成本控制；
- 执行证据；
- 最终结果审计。

长期会走向跨平台执行：在用户授权和平台规则范围内，像可靠的人类助理一样操作网站和电脑，完成阅读、收集、比较、填写、下载、上传、中断恢复和跨平台交接。不以绕过验证码、安全限制、平台权限或反自动化规则为产品目标。

## 核心原则

### 极简
- 功能少而精，每个功能都经高频使用验证
- 能用一个按钮解决的，不用两个

### 可组合
- 脚本之间可以串联，形成工作流
- 输入输出标准化，像搭积木一样

### 本地优先
- 用户数据和任务证据本地优先保存
- 模型请求可以通过 AI Workbench 自控 Managed Proxy 调用
- 真实模型 key 不进入用户电脑、安装包或员工配置
- 离线时保留本地记录和中文未就绪说明

## 产品战略 v2(2026-07-18)

- 使用与结果必须双优：页面好看但办不成事没有竞争力，能办事但难用也没有竞争力，两者都好才有资格上牌桌。
- 核心竞争力=零门槛：市面同类产品（如 Hermes One 连手机版）功能强但需要配置知识；我们把全部功能内置到工作台内部，用户只看到一个对话框，一句话完成一切，打破使用壁垒。
- 功能对标策略：同行已把功能前端展示出来（设备管理/通道绑定/任务编排），照着做，但客户永远只见一个页面。
- 最终标准：能不能办成事。工作台的意义=把人的话传给模型和 Agent，中间消化一切摩擦（环境/记忆/卡壳/壁垒），交付简单的结果；人只负责判断和决策。
- 节奏观：不急变现，先做正确的事、拿到真实反馈，持续上线持续调试，铁杵磨成针。

## 功能规划

### 已完成

- Windows 安装、启动和卸载；
- 无用户 Key 真实模型调用；
- Cloudflare Managed Proxy；
- v0.4.6 公开 Alpha Release。

### 近期

- 产品资产备份和电脑清理审计；
- 首屏 3-5 条示例指令；
- 反馈入口和安全告知；
- 3-5 名真实用户测试；
- 只修影响真实使用的问题。

### 下一阶段

- 长期记忆；
- 任务历史和状态卡；
- 质量检查层；
- 自动任务拆解和分配；
- 多模型、多 Agent 调度。

### 长期

- 跨网站和本机执行；
- 长任务中断恢复；
- 信息收集与研究工作流；
- 全球多语言、区域模型和合规适配。

## 不做

- ❌ 传统 SaaS 式复杂后台和多人管理系统
- ❌ 多人协作
- ❌ 复杂 UI
- ❌ 任何与"卖钱"有关的功能
- ❌ 绕过验证码、安全限制、平台权限或反自动化规则
